# XMT Billing System — End-to-End Flow Guide

> **Audience:** Manager, QA, and developers  
> **Purpose:** End-to-end walkthrough for testing actual runtime flow from setup to closure, with clear trigger type per step.
> **Primary Source:** `XMT___Billing_System.ds`

---

## 1) How to Read This Document

### Trigger Labels

- **[Manual]** User must create/edit a record or click an action/transition.
- **[On Save]** User saves a form and workflow runs automatically.
- **[Schedule]** System runs automatically by configured time/date.

### Test Outcome Labels

- **Pass** Expected output exists and values/status match.
- **Fail** Output missing, incorrect value/status, or runtime error.
- **Blocked** Cannot continue due to dependency/data/config issue.

---

## 2) System-Level Flow Map

1. **[Manual]** Prepare master data and customer context (`Customer_Profile`, `Organization_Settings`, `Tax`, `Payment_Term`).
2. **[Manual]** Create `Tenancy`.
3. **[Manual]** Create `Subscription` with line items.
4. **[On Save]** Subscription workflow initializes billing fields and may create invoice immediately in near-billing conditions (`Handle_Subscription_Submi1`).
5. **[Schedule]** Daily `Subscription_Schedule` creates recurring invoices when billing date criteria match.
6. **[Manual]** User processes invoice lifecycle (approve/send/payment/conversion actions).
7. **[On Save]** Payment workflows update `Amount_Paid`, `Amount_Due`, and blueprint stages on related documents.
8. **[Schedule]** Reminder and overdue schedules continue post-send (`W1st_Remainder_Invoice_Due`, `Final_Invoice_Reminder`, `Change_status_to_Overdue_`).

## 4) End-to-End Test Flows

## Flow A — Recurring Billing (Primary Happy Path)

### Preconditions

- Active `Customer_Profile` exists.
- Required defaults and lookups exist (`Organization_Settings`, tax, payment term, accounts).
- Tester can access forms and blueprint transitions.

### Steps

1. **[Manual]** Create `Tenancy` with billing cycle, `Bill_For`, `Bill_Every`, payment term, and payable account.
   - Verify: record saves, required fields validated.
2. **[On Save]** Tenancy workflows auto-generate account number if missing and run validation logic.
   - Verify: `Account_Number` is generated when empty.
3. **[Manual]** Create `Subscription` linked to the tenancy and add line items.
   - Verify: line quantities and unit prices are valid (> 0).
4. **[On Save]** Subscription workflow (`Handle_Subscription_Submi1`) initializes billing dates and may create invoice immediately for near-billing conditions.
   - Verify: `Latest_Billing_Date`, `Next_Billing_Date`, `Final_Billing_Date` are populated as expected.
5. **[Schedule]** `Subscription_Schedule` runs daily and creates `Invoice` for eligible subscription records.
   - Verify: invoice is created with expected header and line totals.
6. **[Manual]** Open invoice and execute lifecycle transitions (send for approval, approve, send).
   - Verify: stage moves as expected in `Invoice_Blueprint`.
7. **[Manual]** Use invoice record action to open payment (`Record_Payment_Invoice`) and submit `Payment_Received`.
   - Verify: payment document created with correct references.
8. **[On Save]** Payment success workflow updates target invoice balances and stage.
   - Verify: `Amount_Paid`, `Amount_Due`, and stage (`Partially Paid`/`Paid`) are correct.
9. **[Schedule]** For unpaid invoices, reminder and overdue schedules continue to run.
   - Verify: reminder email behavior and overdue transition occur according to due-date timing.

### Pass Criteria

- Invoice is generated by on-save path and/or schedule path at the correct time.
- Invoice lifecycle reaches send/payment states without data inconsistency.
- Payment updates financial fields and document stages correctly.

## Flow B — Pro Forma Deposit Path

### Steps

1. **[Manual]** Create `Pro_Forma_Invoices`.
2. **[On Save]** Number generation and initialization workflows run.
3. **[Manual]** Process approval/sending transitions.
4. **[Manual]** Trigger deposit payment via `Record_Payment_Deposit1`.
5. **[On Save]** `Payment_Received` success workflow updates `Pro_Forma_Invoices` paid/due amounts and stage.

### Pass Criteria

- Pro forma moves through expected lifecycle stages.
- Deposit payment updates amount and stage outcomes correctly.

---

## Flow C — Exception Path: Credit, Debit, Refund

### C1) Invoice to Credit Note and Apply Credit

1. **[Manual]** From invoice, trigger `Convert_To_Credit_Note`.
2. **[On Save]** Credit note number/derived fields initialize.
3. **[Manual]** Complete credit note approval lifecycle.
4. **[Manual]** Trigger `Apply_Credits_to_Invoices`.
5. **[On Save]** Credit application updates target invoice credit fields and due amount.

### C2) Invoice to Debit Note and Payment

1. **[Manual]** From invoice, trigger `Convert_To_Debit_Note1`.
2. **[On Save]** Debit note defaults/number generation run.
3. **[Manual]** Approve/send debit note.
4. **[Manual]** Record payment against debit note.
5. **[On Save]** Payment workflow updates debit note amount and stage.

### C3) Credit Note to Refund Note

1. **[Manual]** From credit note, trigger `Create_Refund_Note_From_C`.
2. **[On Save]** Refund note number generation and validations execute.
3. **[Manual]** Submit/complete refund business process.
4. **[On Save]** Related credit balance synchronization is applied.

### Pass Criteria

- Conversion actions copy core references and totals correctly.
- Credit/debit/refund downstream balances and stage changes are consistent.

## Flow D — `Payment_Received` End-to-End (Dedicated)

This flow isolates payment behavior so manager/QA can test payment posting and reversal without running the full chain each time.

### D1) Payment Against `Invoice`

1. **[Manual]** Open target invoice and trigger `Record_Payment_Invoice`.
2. **[Manual]** Confirm prefilled `Payment_Received` fields (customer, reference, amount context).
3. **[Manual]** Submit `Payment_Received` with full payment amount.
4. **[On Save]** Payment success workflow updates `Invoice.Amount_Paid` and `Invoice.Amount_Due`.
5. **[On Save]** Workflow/blueprint stage update routes invoice to `Paid` (or `Partially Paid` for partial allocations).

Verify:
- `Payment_Received` record links to the correct invoice.
- Invoice `Amount_Paid` increments by posted amount.
- Invoice `Amount_Due` decreases correctly.
- Invoice stage reflects resulting balance (`Paid`/`Partially Paid`).

### D2) Payment Against `Debit_Note`

1. **[Manual]** Open target debit note and trigger payment action (`Record_Payment_Invoice` context is reused on debit note entries in DS).
2. **[Manual]** Submit `Payment_Received` with debit-note allocation.
3. **[On Save]** Payment workflow updates `Debit_Note.Amount_Paid` and `Debit_Note.Amount_Due`.
4. **[On Save]** Stage updates to `Paid` or `Partially Paid` (or returns to `Sent`/`Overdue` based on due-date context when applicable).

Verify:
- Payment line is allocated to the intended debit note.
- Debit note balances and stage transition match posted amount and due-date state.

### D3) Payment Against `Pro_Forma_Invoices` (Deposit)

1. **[Manual]** Open target pro forma invoice and trigger `Record_Payment_Deposit1`.
2. **[Manual]** Submit `Payment_Received` in deposit path.
3. **[On Save]** Payment workflow updates `Pro_Forma_Invoices.Amount_Paid` and `Amount_Due`.
4. **[On Save]** Stage updates to `Paid` or `Partially Paid` (or `Sent`/`Overdue` depending on outstanding amount and due-date context).

Verify:
- Deposit payment references the correct pro forma record.
- Pro forma amount and stage updates are consistent with allocation.

### D4) Payment Reversal / Deletion Check

1. **[Manual]** Delete a posted `Payment_Received` record used in D1/D2/D3.
2. **[On Save]** Delete workflow reverts linked document balances and removes linked payment journal impact.

Verify:
- Target document `Amount_Paid` and `Amount_Due` are recalculated correctly after deletion.
- Stage is re-evaluated correctly after reversal.

### Pass Criteria

- Posting and reversal both keep balances and stages consistent across invoice, debit note, and pro forma targets.
- No orphan allocation/journal side effects remain after payment deletion.
